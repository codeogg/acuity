# 运营端模拟（Impersonation）功能 —— 详细设计文档

版本：V1.0　　状态：待评审　　适用范围：运营端（Admin）× 医生端（Doctor App）

---

## 一、背景与目标

运营/客服人员在处理诊所、医生相关工单时，经常需要"以医生的视角"查看甚至协助操作系统数据（如帮助排查排班异常、协助录入数据）。为了在**支持效率**与**数据安全/合规**之间取得平衡，设计"模拟"功能，核心原则：

1. **透明不隐藏**：模拟行为对医生可见（事后通知），不做偷偷摸摸的静默访问。
2. **权限收窄**：默认最小权限（检视/只读），代理（读写）需要显式选择并二次确认。
3. **全程留痕**：模拟期间的所有访问/操作均需审计，操作人（运营）与被模拟人（医生）身份不得混淆。
4. **有限时效**：模拟不是永久授权，必须有过期机制，杜绝"忘记退出"导致的权限滞留。
5. **架构隔离**：运营端与医生端认证体系互不侵入，医生的真实登录流程不受模拟功能影响。

---

## 二、名词定义

| 术语 | 说明 |
|---|---|
| 操作人 / Operator | 发起模拟的运营/客服人员，拥有运营端账号 |
| 被模拟人 / Subject（Doctor） | 被查看或代理操作的医生 |
| 检视模式 / View Mode | 只读模式，操作人只能查看数据，无法提交/修改任何内容 |
| 代理模式 / Proxy Mode | 读写模式，操作人可代医生编辑并提交数据，所有操作均被记录并归因于操作人 |
| 模拟会话 / Impersonation Session | 一次模拟行为的完整生命周期记录，从发起到结束/过期 |
| 模拟令牌 / Impersonation Token | 一次性、短期有效的签名令牌，用于运营端跳转进入医生端时建立会话，不涉及医生密码 |

---

## 三、总体架构

运营端（admin.xxx.com）与医生端（doctor.xxx.com）是**两个独立的前端应用**，各自拥有独立的登录入口与认证体系，仅在后端**数据服务层**共享同一套业务数据。模拟功能是连接两端的一条"临时授权通道"，不侵入任一方的核心认证逻辑。

```
运营端                                    医生端
┌─────────────┐   1.发起模拟(选诊所/医生/模式)   ┌─────────────┐
│ 张三(operator)│ ───────────────────────────→ │ (无操作)      │
│              │   2.生成一次性模拟令牌          │              │
│              │   3.新标签页携带令牌跳转 ───────→│ 4.令牌校验     │
│              │                              │ 5.建立带上下文  │
│              │                              │   的医生端会话  │
│              │                              │ 6.顶部横幅提示  │
│              │ ←────── 8.退出模拟（双端均可发起）│ 7.浏览/操作    │
└─────────────┘                              └─────────────┘
        │                                            │
        └──────────────→ 审计日志（统一记录）←────────────┘
```

**关键设计决策：模拟不等于"登录医生账号"**。操作人全程使用自己的运营身份，系统通过"身份上下文（operator_id + target_doctor_id + mode）"挂载数据访问范围，而不是真正获取医生的账号密码或 Token。这保证了：
- 审计链条不断裂，所有操作可唯一追溯到运营人员本人；
- 不会占用/顶替医生本人的登录会话；
- 检视/代理的权限收窄可以在同一套鉴权代码里统一处理，不依赖"伪装成医生"后系统是否自觉遵守限制。

## 四、数据模型

### 4.1 模拟会话表 `impersonation_sessions`

不在医生表上加字段记录模拟状态，而是单独建表记录每一次模拟的完整生命周期，理由：医生表是静态业务实体，模拟是有生命周期的会话行为，混用会导致并发脏数据、无法追溯历史。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT | 主键 |
| clinic_id | VARCHAR | 诊所ID（模拟范围按"诊所+医生"联合维度隔离，见4.3） |
| doctor_id | BIGINT | 被模拟医生 |
| operator_id | BIGINT | 发起模拟的运营人员，审计核心字段，不可省略 |
| mode | ENUM | view（检视）/ proxy（代理） |
| reason | VARCHAR | 模拟原因/关联工单号 |
| status | ENUM | active / ended / expired |
| started_at | DATETIME | 开始时间 |
| ended_at | DATETIME | 结束时间（主动结束或过期均记录） |
| expire_at | DATETIME | 强制过期时间点 |
| last_active_at | DATETIME | 最近一次操作时间，用于空闲超时判断（代理模式） |
| doctor_notified_at | DATETIME | 医生端是否已展示通知，NULL 表示未通知 |
| doctor_acknowledged_at | DATETIME | 医生是否已点击"知晓"确认 |
| created_at | DATETIME | 记录创建时间 |

### 4.2 模拟令牌（不落库或短期缓存，非核心数据表）

一次性签名令牌（JWT 或等价机制），有效期建议 **5 分钟内**，包含 `session_id / operator_id / doctor_id / clinic_id / mode`，验证时需校验签名、有效期、是否已被使用（防重放），使用后立即失效。

### 4.3 唯一性约束

```sql
UNIQUE(clinic_id, doctor_id) WHERE status = 'active'
```

按"诊所 + 医生"联合维度隔离，而非只按医生维度。原因：同一医生可能在多家诊所出诊，模拟范围应限定在发起模拟时所在的那一家诊所，不应默认放大到该医生名下的全部诊所数据，符合最小必要授权原则。若需要查看该医生在另一诊所的数据，需在该诊所详情页重新发起一次独立的模拟会话。

## 五、核心流程

### 5.1 发起模拟

1. 运营人员在运营端诊所详情页 → "模拟" Tab → 选择医生 → 选择模式（检视/代理）。
2. 代理模式需二次确认弹窗（"所有操作均被记录并归因于我"），检视模式可直接进入。
3. 后端校验该 `(clinic_id, doctor_id)` 是否已有 active 会话：
   - 无：创建新会话记录，生成模拟令牌。
   - 有：直接复用现有会话生成新令牌（避免重复点击导致报错或产生多条会话）。
4. 前端携带令牌以新标签页方式跳转至医生端专用入口（非医生登录页）。

### 5.2 医生端令牌校验入口

医生端提供独立接口 `POST /doctor/session/impersonation-entry`，**不复用** `/doctor/login`：

- 校验令牌签名、有效期、是否已被使用；
- 校验对应 `impersonation_sessions` 记录状态确实为 active；
- 建立挂载 `impersonation_context` 的医生端业务 Session/Token；
- 全程不涉及医生账号密码，不占用医生本人的登录名额，不会顶掉医生正在使用的会话。

由于诊所范围已在令牌生成时锁定，**运营进入医生端时不会出现"选诊所"步骤**，与医生本人登录后需要手动选诊所是两条不同的路径，互不影响。

### 5.3 会话期间的页面表现

- 页面顶部常驻横幅：模式颜色区分（如蓝色=检视 / 紫色=代理），文案示例："检视模式中 —— 正在查看 陈志明 医生数据"。
- 浏览器标签页标题/图标同步变化，避免与操作人自己的运营端标签页混淆。
- 横幅上提供"退出模拟"按钮。
- 检视模式下，所有写操作入口（按钮/表单提交）前端隐藏或禁用（体验层面，非安全边界）。

### 5.4 退出模拟

- 点击"退出模拟" → 调用 `POST /admin/impersonation/end` → 会话状态置为 `ended`，记录结束时间。
- 前端**先清除本地 Token**（权限收回优先级最高），再尝试关闭标签页。
- 若浏览器拦截 `window.close()`（脚本非自身打开的标签页常见此限制），展示"模拟已结束，可关闭此标签页"的提示页，不强依赖自动关闭。
- 运营端原页面同样提供"结束此次模拟"入口，防止医生端标签页异常关闭（崩溃/被关）导致无法主动结束。

### 5.5 超时兜底机制

用户不点击退出按钮直接关闭浏览器是大概率场景，因此不能只依赖前端主动上报，必须有服务端兜底：

| 场景 | 机制 |
|---|---|
| 绝对超时 | 检视模式 30–60 分钟，代理模式 15–30 分钟（更短），到期由后台任务/懒惰查询将 `active` 置为 `expired` |
| 空闲超时（仅代理模式） | 超过一定时间（如 10 分钟）无任何操作请求，自动结束会话，比绝对超时更早收回高风险权限 |
| 续期 | 允许续期，但需重新走一次授权确认（代理模式重新弹窗确认），不可静默无感延长 |

### 5.6 医生端事后通知

- 医生本人正常走 `/doctor/login`（不改动核心鉴权流程），登录成功后**异步、非阻塞**调用独立接口 `GET /doctor/support-access/pending` 查询是否存在未读模拟记录。
- 查询范围需覆盖该医生名下**所有绑定诊所**，不能仅限当前选中诊所。
- 该查询失败或超时不得影响医生正常登录，最多导致通知延迟展示。
- 医生确认后回写 `doctor_acknowledged_at`。
- 若同一时段被多名运营模拟过，按每次会话独立展示（因 operator_id、clinic_id、时间均不同，通知天然是多条，不合并）。

## 六、权限控制（检视 / 代理 / 禁止）

### 6.1 双注解设计（正交，非耦合）

访问控制与审计详细度是两个独立维度，不合并成一套注解：

```java
// 维度一：访问控制
@ImpersonationAccess(READ_ONLY)   // 检视模式下放行
@ImpersonationAccess(MUTATING)    // 检视模式下拦截；代理模式下放行（默认值，未标注按此处理）
@ImpersonationAccess(FORBIDDEN)   // 检视/代理模式下均拦截（如改密码、账号注销、绑卡信息变更等高危操作）

// 维度二：审计详细度（可选叠加）
@AuditSensitive(resource = "patient_record", level = HIGH)
```

**默认拒绝原则**：接口未显式标注时，一律按 `MUTATING` 处理。新增接口若开发遗漏标注，代价是检视模式下多拦截了一个本该放行的只读接口（体验问题），而不会出现"漏拦截导致检视模式下能改数据"（安全事故），方向不可颠倒。

### 6.2 不能仅凭 HTTP Method 判断读写

部分接口是"伪装成 GET 的写操作"（如 `GET /reports/export?send_email=true` 触发发信副作用），因此读写属性必须由接口显式声明，不能靠约定猜测。

### 6.3 权限判断层级

| 层级 | 内容 | 定位 |
|---|---|---|
| 前端 | 检视模式隐藏/禁用写操作按钮 | 体验优化，**不是安全边界** |
| 后端中间件（切面） | 依据 `@ImpersonationAccess` 判定放行/拦截 | **真正的安全边界，必须实现** |
| 兜底 | 未标注接口默认 MUTATING | 防止漏标导致越权 |

## 七、审计日志设计

### 7.1 总原则

不依赖人工维护"哪些接口需要记录"的清单（必然会漏），而是通过全局中间件默认拦截全部处于模拟会话中的请求，再按操作性质自动分级，控制记录深度。

### 7.2 切面执行顺序（顺序不可颠倒）

```
1. 判断当前请求是否处于活跃模拟会话中
   否 → 直接放行，跳过以下全部逻辑（医生本人正常操作不受影响）

2. 【基础日志】无条件记录（此步骤必须在拒绝判断之前执行）：
   session_id / operator_id / doctor_id / mode / path / method / http_status / timestamp / ip / latency_ms
   —— 即使后续被拒绝，"曾尝试执行此操作"本身也是重要审计信息，不可省略

3. 【访问控制判断】读取 @ImpersonationAccess（无注解按 MUTATING）：
   FORBIDDEN → 直接 403，两种模式均拒绝
   MUTATING  → view 模式 403 拒绝；proxy 模式放行
   READ_ONLY → 放行

4. 【敏感标记】通过第3步后，若接口带 @AuditSensitive，追加敏感标记 + 具体访问的资源ID

5. 【写操作 diff】通过第3步后，若为 MUTATING 类型，记录操作前后字段差异
```

### 7.3 记录深度分级

| 操作类型 | 记录内容 |
|---|---|
| 普通只读（GET） | 基础日志字段，不存响应体，避免患者数据在日志中堆积 |
| 敏感只读（如查看完整病历） | 基础日志 + 资源ID + 敏感标记，仍不存整份返回内容 |
| 写操作（POST/PUT/PATCH/DELETE） | 基础日志 + 请求参数 + 操作前后字段 diff |
| 禁止类操作（FORBIDDEN） | 仅记录"尝试执行但被拒绝"，无需 diff |

### 7.4 接口语义标注（可选，非安全底线）

可为接口附加轻量注解标注业务含义（如 `@AuditAction(name="查看患者病历")`），用于将日志展示为可读的自然语言，供管理员查阅审计记录时理解操作内容。**该标注只是展示层的加分项**，即使遗漏，第一层中间件仍会记录原始路径与方法，不影响审计完整性。

## 八、需要新增/调整的接口清单

| 接口 | 归属 | 说明 |
|---|---|---|
| `POST /admin/impersonation/start` | 运营端 | 发起模拟，入参 clinic_id、doctor_id、mode、reason；返回令牌与跳转链接 |
| `POST /admin/impersonation/end` | 运营端 | 主动结束模拟会话 |
| `GET /admin/impersonation/session` | 运营端 | 查询当前诊所/医生是否已有活跃会话 |
| `POST /doctor/session/impersonation-entry` | 医生端 | 校验模拟令牌，建立带上下文的医生端会话，**独立于医生登录接口** |
| `GET /doctor/support-access/pending` | 医生端 | 医生登录成功后异步调用，查询未读模拟通知（跨该医生所有绑定诊所） |
| `POST /doctor/support-access/acknowledge` | 医生端 | 医生确认已知晓某条模拟通知 |

## 九、边界场景与处理策略汇总

| 场景 | 处理策略 |
|---|---|
| 同一医生多诊所出诊 | 模拟范围按 `(clinic_id, doctor_id)` 隔离，一次模拟仅限当前诊所数据 |
| 运营重复点击"进入模拟" | 复用已有 active 会话生成新令牌，不重复建会话/不报错 |
| 医生端标签页被意外关闭 | 运营端保留"结束模拟"入口，可反向结束会话；同时依赖 expire_at 兜底 |
| 用户不点退出直接关浏览器 | 绝对超时 + 代理模式空闲超时双重兜底，自动置为 expired |
| 模拟服务/审计服务异常 | 医生登录核心鉴权不受影响，通知查询失败仅导致通知延迟，不阻塞登录 |
| 多名运营先后模拟同一医生 | 医生端通知按会话逐条展示，不合并，可清晰追溯每次访问的操作人与时间 |
| 接口未标注访问控制注解 | 默认按 MUTATING 处理，检视模式下拒绝，避免漏标导致越权 |
| 高危操作（改密码/账号注销等） | 使用 `FORBIDDEN` 标注，检视/代理模式下均禁止，不受模式选择影响 |

## 十、后续实施建议（优先级排序）

1. 建立 `impersonation_sessions` 表 + start/end/session 查询接口。
2. 实现模拟令牌生成与医生端独立入口校验接口（`impersonation-entry`）。
3. 实现全局审计中间件（第一层基础日志，保证不漏）。
4. 实现 `@ImpersonationAccess` 注解与权限拦截切面（安全边界）。
5. 实现写操作 diff 记录与 `@AuditSensitive` 敏感标记（第二、三层）。
6. 实现医生端登录后异步通知查询与确认接口。
7. 实现超时兜底任务（绝对超时 + 代理模式空闲超时）。
8. （可选，体验优化）为接口补充 `@AuditAction` 语义标注，提升日志可读性。
