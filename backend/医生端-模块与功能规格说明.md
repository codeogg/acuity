# 医生端（Doctor App）—— 完整模块与功能规格说明

> 面向AI Coding直接实现。医生端定位：诊所医生在接诊现场/接诊后，快速完成保险单据填报的高频操作工具，核心诉求是"快、准、少打字"。

---

## 0. 信息架构总览

```
/doctor
├── /                          首页工作台
├── /new-claim                 新建填报入口(选保司)
│   └── /new-claim/[companyId] 选模板
├── /claims/new                填报流程主体(病历输入→AI识别→核对→生成→打印)
│   ├── ?step=record            病历输入
│   ├── ?step=review             字段核对
│   └── ?step=preview            PDF预览/打印
├── /claims                    历史记录列表
│   └── /claims/[id]            填报详情
│       └── /claims/[id]/preview  PDF预览打印页(可从详情跳转)
├── /presets                    常用项管理(常用诊断/费用组合)
├── /profile                    个人中心(基本信息/签名/密码)
└── /notifications               通知中心(可选，一期可做成首页内嵌小组件，不单独开页面)
```

**导航方式**：桌面/平板用左侧边栏（工作台/新建填报/历史记录/常用项/我的），移动端用底部Tab（工作台/新建/记录/我的，常用项收进"我的"二级菜单）。

## 1. 模块一：首页工作台（Home / Workbench）

### 1.1 页面内容清单

| 区块 | 内容 | 数据来源 |
|---|---|---|
| 问候语区 | "早上好，{医生姓名}医生" + 诊所名称，根据当前时间段自动切换"早上好/下午好/晚上好" | 当前登录用户信息(JWT解出) |
| 主操作按钮 | "新建填报"，全局最高优先级CTA，点击进入模块2流程 | — |
| 统计卡片(3张) | 今日已填报数、待确认草稿数、本月累计数 | 见1.2接口 |
| 未完成草稿列表 | 状态为`DRAFT`或`AI_FILLED`的记录，最多展示5条，超过5条显示"查看全部" | 见1.2接口 |
| 快捷开始 | 医生最近使用过的"保司+模板"组合(去重取最近2-4个)，点击直接跳过选保司/选模板两步，进入病历输入页 | 见1.2接口 |
| 最近填报记录 | 最近10条记录(不限状态)，含患者名、保司名、时间、状态徽标，点击进入详情 | 见1.2接口 |
| (可选)公告条 | 系统通知，如"友邦保险新增模板已发布"，可关闭 | 见模块6 |

### 1.2 接口设计

```
GET /api/doctor/home/overview

Response:
{
  "greeting_name": "陈大文",
  "clinic_name": "仁济诊所（尖沙咀分店）",
  "stats": {
    "today_count": 6,
    "pending_draft_count": 2,
    "month_total_count": 87
  },
  "unfinished_drafts": [
    {
      "submission_id": 1001,
      "patient_name": "陈小明",
      "company_name": "友邦保险",
      "template_name": "门诊索偿申请表",
      "status": "AI_FILLED",
      "status_label": "待核对字段",
      "updated_at": "2026-07-04T09:12:00+08:00"
    }
  ],
  "quick_start_shortcuts": [
    { "company_id": 1, "company_name": "安盛保险", "template_id": 101, "template_name": "门诊索偿申请表" },
    { "company_id": 2, "company_name": "友邦保险", "template_id": 205, "template_name": "门诊索偿申请表" }
  ],
  "recent_claims": [
    {
      "submission_id": 998,
      "patient_name": "王丽",
      "company_name": "安盛保险",
      "status": "PRINTED",
      "status_label": "已打印",
      "created_at": "2026-07-04T10:20:00+08:00"
    }
  ]
}
```

**后端实现要点：**
- `unfinished_drafts`：查询 `claim_submission` where `doctor_id = current_doctor` and `status in ('DRAFT','AI_FILLED')`，按`updated_at`倒序限5条
- `quick_start_shortcuts`：查询该医生历史`claim_submission`按`(company_id, template_id)`分组统计使用次数，取最近使用的去重组合（可以用"最近一次使用时间"排序，而不是"使用频率"，因为医生的下一个患者可能刚好又是同一家保司，时间上的"最近"比频率上的"最常用"更贴近真实需求）
- `today_count`/`month_total_count`：按医生+日期范围count，建议做成单条SQL聚合查询，不要拆成多次请求
- 整个页面**只用这一个接口**加载，避免首页多个区块各自发请求造成的加载闪烁和竞态问题

### 1.3 交互细节
- 未完成草稿点击后，直接跳转到该草稿当前所处的步骤（`DRAFT`→病历输入页，`AI_FILLED`→字段核对页），不需要医生重新选择
- 统计卡片点击"待确认草稿"卡片，直接跳转到历史记录页并预设筛选条件`status=AI_FILLED`
- 快捷开始点击后，直接创建新的`claim_submission`草稿(带上该company_id/template_id)并跳转到病历输入页，省去模块2的两步选择

## 2. 模块二：新建填报入口（选保司 → 选模板）

### 2.1 页面内容

**Step A：选择保险公司**（`/new-claim`）
- 网格/列表展示当前诊所已启用的保险公司（含LOGO），搜索框可按名称过滤
- 空状态：若诊所未配置任何保司，提示"暂无可用的保险公司，请联系管理员配置"，不给医生自行操作的入口（这是配置权限之外的事）

```
GET /api/doctor/insurance-companies
Response: [
  { "company_id": 1, "company_name": "安盛保险 AXA", "logo_url": "..." }
]
```

**Step B：选择保单模板**（`/new-claim/[companyId]`）
- 列出该保司下、当前诊所已启用、且已发布(`is_active=true`)的模板
- 每个模板卡片展示：模板名称、版本号、"最近使用"标记（若该医生用过这个模板）
- 空状态：若该保司下暂无可用模板，提示"该保险公司暂无可用模板，请联系管理员配置"

```
GET /api/doctor/insurance-companies/{companyId}/templates
Response: [
  { "template_id": 101, "template_name": "门诊索偿申请表", "version": "V2", "last_used_at": "2026-07-01T..." }
]
```

### 2.2 创建填报记录

选定模板后立即创建草稿并跳转，不需要额外"确认"按钮：

```
POST /api/doctor/claims
Body: { "company_id": 1, "template_id": 101 }
Response: { "submission_id": 1050, "submission_no": "SUB20260704001" }
→ 前端跳转至 /claims/new?id=1050&step=record
```

## 3. 模块三：填报流程主体

这是医生端使用频率最高、体验要求最高的核心模块，按 `claim_submission.status` 状态机分三个子页面/子步骤。

### 3.1 Step 1：病历输入（`status=DRAFT`）

**页面内容：**
- 顶部：患者姓名快速输入框（可选，先填会让后续列表识别更方便，不填也能继续）
- 大文本框：病历文本粘贴/输入区，占据页面主要空间
- 底部：字数统计 + "开始AI识别"主按钮

**交互细节：**
- 文本框支持粘贴富文本时自动转纯文本（避免粘贴带格式内容导致AI识别异常）
- 草稿自动保存：文本框内容变化后，防抖2秒自动调用保存接口，避免医生中途被打断（接诊现场很常见）而丢失已输入内容
- "开始AI识别"点击后，按钮变为loading态，文案改为"AI识别中，预计需要几秒"，同时禁用重复点击

```
PUT /api/doctor/claims/{id}/draft
Body: { "patient_name": "陈小明", "medical_record_text": "..." }
Response: { "saved_at": "..." }

POST /api/doctor/claims/{id}/extract
（内部触发AI识别，见AI识别模块规格文档3.1节，此处只是从医生端UI角度描述调用时机）
Response: { extracted_fields: {...}, process_time_ms }
→ 成功后 status 变为 AI_FILLED，前端自动跳转 step=review
→ 失败(AI服务不可用)：不阻塞，前端提示"AI识别暂时不可用"，
   直接进入 step=review，所有字段留空，医生手动填写
```

### 3.2 Step 2：字段核对（`status=AI_FILLED`）—— 核心交互页面

**页面布局：左右分栏**
- 左侧：病历原文只读展示（方便医生对照原文核对）
- 右侧：AI识别出的结构化字段表单，按 `field_domain` 分组展示（患者信息/诊断信息/治疗信息/费用信息...）

**每个字段的展示规则：**
| 字段数据类型 | 渲染控件 |
|---|---|
| text | 文本输入框 |
| date | React Aria日期选择器(分段输入) |
| enum | 单选按钮组，选项=`enum_options` |
| multi_select | 多选勾选框组 |
| number | 数字输入框，右对齐+等宽字体 |
| boolean | 开关/勾选框 |

**置信度视觉提示：**
- `confidence >= 0.8`：正常显示，无特殊标记
- `0.5 <= confidence < 0.8`：字段边框用警示色(warning)，右侧小图标+tooltip"AI识别置信度中等，建议核对"
- `confidence < 0.5` 或 `value为null`：字段边框用危险色(danger)，标注"AI未能识别，请手动填写"，且该字段自动获得初始焦点顺序优先（表单顶部优先展示需要处理的字段，或者提供"仅看待核对项"筛选开关）

**操作方式：**
- 支持Tab键在字段间快速跳转（医生高频操作习惯）
- 修改任意字段值，实时对比原AI识别值，若不同则该字段打上"已修改"小标记（用于后续`claim_field_change_log`记录）
- 底部主按钮："确认并生成保单"

```
PUT /api/doctor/claims/{id}/fields
Body: { "final_field_values": { "patient_name_cn": "陈小明", "diagnosis": "...", ... } }
副作用：后端diff对比ai_raw_result，写入claim_field_change_log

POST /api/doctor/claims/{id}/confirm
→ 校验is_required的标准字段是否都有值
→ 校验失败：返回缺失字段列表，前端定位并高亮对应输入框，不允许继续
→ 校验成功：status → CONFIRMED，前端跳转 step=preview
```

### 3.3 Step 3：PDF预览与打印（`status=CONFIRMED`）

**页面内容：**
- 内嵌PDF预览（pdf.js渲染生成好的保单PDF）
- 操作按钮：打印、下载、返回修改（退回Step 2重新核对，允许再次修改后重新生成）

```
POST /api/doctor/claims/{id}/generate-pdf
Response: { "pdf_url": "..." }

（前端展示PDF，用户点击打印后）
POST /api/doctor/claims/{id}/mark-printed
Response: { status: "PRINTED" }
```

**交互细节：**
- 若医生点"返回修改"，允许直接跳回Step 2（不需要重新走病历输入），`status`退回`AI_FILLED`
- 打印动作触发系统打印对话框(`window.print()`或PDF直接走浏览器打印)，标记打印完成不依赖浏览器打印是否真的物理打印成功（这个无法准确检测），是医生主动点击"已打印"来确认

## 4. 模块四：历史记录

### 4.1 列表页（`/claims`）

**筛选项：** 患者姓名(模糊搜索)、状态、保险公司、日期范围
**列表每行展示：** 患者姓名、保司+模板名、创建时间、状态徽标
**操作：** 点击进入详情；列表行内快捷操作"重新打印"（针对`PRINTED`状态，直接调打印不需要进详情页）

```
GET /api/doctor/claims?patient_name=&status=&company_id=&date_from=&date_to=&page=&page_size=20
Response: {
  "items": [...],
  "total": 145,
  "page": 1
}
```

### 4.2 详情页（`/claims/[id]`）

**内容：**
- 基本信息：患者姓名、保司、模板(含版本)、创建/更新时间、当前状态
- 字段值展示（只读，`final_field_values`按字段域分组展示）
- 操作区（按当前status显示不同操作）：
  - `DRAFT`/`AI_FILLED`：继续填报（跳回对应步骤）
  - `CONFIRMED`/`PRINTED`：查看/重新打印PDF、复用到其他模板（见4.3）
  - 任意状态：作废（需二次确认）

```
GET /api/doctor/claims/{id}
Response: {
  "submission_no": "...",
  "patient_name": "...",
  "company_name": "...",
  "template_name": "...",
  "template_version": "...",
  "status": "...",
  "final_field_values": {...},
  "generated_pdf_url": "...",
  "created_at": "...",
  "updated_at": "..."
}

POST /api/doctor/claims/{id}/cancel
Body: { "reason"?: string }
```

### 4.3 复用到其他保司模板

同一患者同一次就诊，需要出具多份不同保司的保单时使用：

```
POST /api/doctor/claims/{id}/reuse-for-template
Body: { "new_company_id": 2, "new_template_id": 205 }
Response: { "submission_id": 1051, "prefilled_fields": {...} }
→ 前端跳转到新submission的 step=review 页面，
  已从原记录的final_field_values中取交集字段预填，
  缺失的字段（新模板需要但原记录没有的标准字段）留空/触发针对性AI提取
```

**入口位置**：填报详情页 + Step3预览页，都提供"生成其他保司的保单"按钮，避免医生重新粘贴一遍病历。

## 5. 模块五：常用项管理（效率优化，建议二期做但优先级较高）

### 5.1 功能说明

医生高频重复的诊断描述、费用组合（比如"上呼吸道感染 + 诊金150 + 药费80"这种常见套餐），保存为个人常用项，在Step 2字段核对页面可以一键套用，减少重复打字。

### 5.2 页面内容（`/presets`）

- 列表：医生本人保存的常用项，每条展示"名称+关联字段值摘要"
- 新增/编辑：选择要保存的字段范围(如只存诊断+费用相关字段，不存患者个人信息)，命名保存

```
GET  /api/doctor/presets
POST /api/doctor/presets
Body: { "preset_name": "普通感冒套餐", "field_values": { "diagnosis": "...", "consultation_fee": 150 } }
DELETE /api/doctor/presets/{id}
```

### 5.3 在核对页面的调用方式

Step 2核对页面顶部提供"套用常用项"下拉，选中后把对应字段值批量填入表单（仍可再次手动修改，不是锁定值）。

## 6. 模块六：个人中心

### 6.1 页面内容（`/profile`）

- 基本信息展示：姓名、医委会注册编号、所属诊所（只读，医生自己不能改，需管理员改）
- 电子签名：展示当前签名图片，支持重新上传替换（用于PDF自动盖签名）
- 修改密码：旧密码+新密码+确认新密码

```
GET  /api/doctor/profile
PUT  /api/doctor/profile/signature   (multipart上传图片)
PUT  /api/doctor/profile/password
Body: { "old_password": "...", "new_password": "..." }
```

## 7. 模块七：通知（可选，轻量实现）

一期不需要单独的通知中心页面，用首页工作台顶部的**可关闭公告条**即可满足"模板更新/保司变更"这类信息告知需求：

```
GET /api/doctor/notifications/active
Response: [
  { "id": 1, "message": "友邦保险新增《门诊索偿申请表》模板已发布", "level": "info" }
]
POST /api/doctor/notifications/{id}/dismiss   # 该医生标记已读/关闭，不再展示
```

## 8. 跨模块的通用规则

### 8.1 数据隔离
所有 `/api/doctor/**` 接口强制注入 `clinic_id`（来自JWT），历史记录、字段核对等所有查询自动带上该过滤条件，防止跨诊所越权（复用之前功能规格文档模块5已定义的规则，此处不重复）。

### 8.2 移动端/平板适配要点
- Step 2字段核对页在窄屏(平板竖屏/手机)下，"病历原文"区域改为可折叠的抽屉，默认收起，医生需要对照时点开，避免和表单争夺屏幕空间
- 底部Tab导航在≤768px宽度下启用，桌面宽屏用左侧边栏
- PDF预览页在移动端优先保证"打印/下载"按钮可触达，PDF本身可捏合缩放查看

### 8.3 离开页面保护
Step 1/Step 2 编辑中如果医生尝试离开页面（浏览器返回/关闭标签），若有未保存的修改（对比最后一次自动保存时间点），弹出确认提示，防止误操作丢失核对进度。

### 8.4 加载与空状态文案（遵循此前定义的写作语气规范）
- 空状态："还没有填报记录，点击右上角开始新建" 而非"暂无数据"
- AI识别失败："AI识别暂时不可用，你可以直接手动填写下面的字段" 而非"系统错误"
- 必填校验失败："还有 3 个必填字段未填写" + 定位到具体字段，而非笼统的"请完善信息"

## 9. 页面与状态机对照速查表

| 页面路径 | 对应`claim_submission.status` | 允许的操作 |
|---|---|---|
| `/` | — | 查看概览、发起新建 |
| `/new-claim`, `/new-claim/[companyId]` | 创建前 | 选保司、选模板 |
| `/claims/new?step=record` | DRAFT | 输入病历、触发AI识别 |
| `/claims/new?step=review` | AI_FILLED | 核对/修改字段、确认 |
| `/claims/new?step=preview` | CONFIRMED | 预览、打印、返回修改 |
| `/claims`（列表） | 任意 | 筛选查看历史 |
| `/claims/[id]`（详情） | 任意 | 查看详情、继续/复用/作废 |

## 10. 建议开发顺序

1. 首页工作台的接口先返回mock/最简实现，UI搭好骨架（因为它依赖其他模块的数据，放最后接真实数据也可以，但页面结构先行）
2. 模块二（选保司/模板）+ 模块三（填报流程三步）—— 这是核心链路，优先完整跑通
3. 模块四（历史记录）
4. 首页工作台接入真实统计接口
5. 模块五（常用项）、模块六（个人中心）、模块七（通知）作为体验优化，放最后
