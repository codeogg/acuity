# 文档信息提取原则性目标

> 本文档提炼自 [`REQUIREMENTS.zh-CN.md`](REQUIREMENTS.zh-CN.md)，规定 CMS 病历到理赔表单的信息提取系统中**不可违背**的原则性目标。  
> 具体量化指标、测试用例与评分流程仍以 `REQUIREMENTS.zh-CN.md` 为准。

---

## 一、总则（四条不可违背的底线）

| 原则 | 含义 |
|------|------|
| **如实提取** | 文档里有的、能确定的，必须提取并标准化 |
| **如实缺失** | 文档里没有的，必须标 `missing`，禁止编造、推断、跨文档搬运 |
| **剔除干扰** | 噪声信息不得进入任何理赔字段 |
| **绑定正确** | 所有字段必须归属正确的病人、正确的就诊 |

**设计准则（理赔场景）：**

> 宁可漏提、低置信、标缺失，也不可填错人、填错就诊、填干扰项、填不存在的数据。

---

## 二、输出完整性

1. **全字段覆盖**：每个标准目标字段必须有明确结果——标准化取值，或 `missing`（并说明原因）。不允许无输出、空着不管。
2. **结构化可审计**：每个字段至少输出 `value`、`status`、`confidence`、`source`。
3. **低质量不等于乱填**：OCR 差、扫描模糊时，下调置信度或标 `low_confidence`，**不得用乱码或猜测值充数**。

### 标准输出契约

```json
{
  "field_code": {
    "value": "标准化后的取值或 null",
    "status": "extracted | missing | low_confidence",
    "confidence": 0.0,
    "source": { "page": 1, "label": "原文标签", "bbox": null }
  }
}
```

---

## 三、真实性（反臆造）

1. 文档中不存在的信息，一律 `missing`；不得推断、补全、从其他文档借用。
2. 打码身份证号（如 `A12XXX6(7)`）必须原样保留，**不得猜测被遮蔽数字**。
3. 必须区分：
   - 字段**本就不存在** → `missing`
   - 字段**可能存在**但 OCR 置信度低 → `low_confidence`
4. 文档明确写「未记录」等，应判为缺失，不得当作有值。

---

## 四、归属正确性

### 4.1 病人归属

- 多病人文档中，所有字段必须绑定到理赔目标病人（如「索償對象」）。
- **绑错病人 = 一票否决**。

### 4.2 就诊归属

- 多就诊文档中，必须绑定本次理赔对应就诊（住院 / 手术 / 预授权），不得误用普通科、药房等无关记录。
- 支持就诊选择时，提取范围**仅限所选就诊**，不得跨就诊串用。
- 全量病历导出时，只提取本次理赔就诊；不得混入慢性病列表、既往诊断/手术、长期用药、化验、疫苗接种、家族/社会史、其他无关就诊。

### 4.3 本次 vs 背景

- 区分本次理赔相关（如阑尾炎发作）与背景慢性病（如糖尿病、高血压）。
- 不得把背景慢性病当作本次理赔诊断或手术。

---

## 五、干扰剔除（零误填）

### 5.1 不得映射到理赔字段的噪声

排队号、预约编号、下次预约日期、会员/积分及等级、营销同意、获知渠道、病历卡颜色、诊室/床位号、网络/路由代码、条形码、药房 SKU、打印人/打印时间、营业时间、访客 WiFi、评价星级、天气、模板/隐私声明（PICS）等。

### 5.2 常见混淆（任一发生 = 一票否决）

| 干扰项 | 绝不能当作 |
|--------|------------|
| 积分 | 会员号 |
| 网络代码 / 系统流水号 / 条形码 | 保单号 |
| 预约编号 | 收据号 |
| 下次预约日期 | 就诊 / 入院日期 |
| 打印人 | 医生 / 病人姓名 |
| Consultation fee / 诊金 / 咨询费 | amount_total（总金额） |

**干扰剔除率要求为 100%**：任何干扰项被填入字段，直接判不通过。

---

## 六、标准化

1. 原文标签（中英文）映射到统一标准字段；中英标签等价（如 診斷 = Diagnosis）。
2. 日期统一为 `YYYY-MM-DD`；歧义格式（如 `MM/DD/YYYY`）须判定或标记歧义，**禁止默默按错误方式解析**。
3. 金额统一为「数值 + 币种（HKD）」。
4. 枚举字段映射到标准值（如 男/女 → Male/Female；病房等级 → Private/Semi-private/Standard/ICU/Day）。
5. 姓名、保单号、收据号等按原文提取；**不得与相似字段混淆**。
6. 医生姓名与注册编号连排时，必须拆分，不得混在一个字段里。

---

## 七、消歧与冲突处理

1. 多值冲突（最终 vs 初步诊断、总额 vs 明细之和）须按规则取权威值（最终优先于初步），并**显性提示冲突**，禁止默默选一个。
2. 跨页表格/金额须正确汇总，不得只取单页。
3. 同一次就诊在多处出现时，须核对归并为同一套取值。

---

## 八、置信度与可复核

1. 每个字段必须有置信度；OCR、打码、低对比度、有冲突的字段，置信度必须低于干净文本层字段。
2. 每个字段必须有来源（至少页码；条件允许时含边界框或原文标签）。
3. 置信度低、检测到冲突、或关键理赔字段缺失时，必须标记需人工复核，**不得假装高置信度过关**。

---

## 九、输入适应性（实现手段，服务于上述目标）

- 自动判别 PDF 有/无文本层，不得默认有文本层。
- 支持多页、多版式（单栏、双栏、表格、发票、信函、传真封面等）。
- 扫描件须 OCR；处理旋转（90°/180°）、倾斜、噪点、低对比度、JPEG 劣化。
- 支持纯中文、纯英文、中英混合及跨语言「标签—取值」配对。
- 盖章、手写批注不得干扰相邻印刷字段；手写不要求转写，但不得误当字段值。

---

## 十、绝对红线（一票否决）

| 红线 | 说明 |
|------|------|
| 绑错病人 | 字段取自非理赔目标病人 |
| 臆造取值 | 文档根本没有的数据被填入字段 |
| 干扰项入字段 | 如积分当会员号、网络代码当保单号 |
| 歧义日期默默解析错 | 高置信度输出错误日期且未标歧义 |
| 打码身份证被补全 | 遮蔽数字被猜测填充 |
| 冲突诊断未区分 | 初步诊断被当作最终诊断高置信输出 |
| 跨就诊串用 | 未选定或非目标就诊的数据用于本次理赔 |
| 既往/慢性数据并入本次理赔 | 背景病史被当作本次诊断/手术 |

---

## 十一、与标准字段库的关系

标准字段定义以 `apps/api/src/seed.py` 及 `REQUIREMENTS.zh-CN.md` 第一节为准。姓名在系统中拆分为 `patient_name_cn` / `patient_name_en`，其余字段对齐 REQUIREMENTS 命名（如 `dob`、`gender`、`hkid`、`diagnosis_text`、`amount_total` 等）。

实现提取逻辑、Prompt、校验、测试时，**必须先满足本文原则，再追求准确率与召回率**。

---

## 十二、11 步流水线架构原则（实现约束）

病历 PDF 提取按 11 步流水线实现（详见 [`病历PDF智能提取_功能文档.md`](病历PDF智能提取_功能文档.md)）：

```
Step1 上传PDF → Step2 PDF预处理 → Step3 OCR → Step4 文档分类
→ Step5 多就诊检测/选择 → Step6 Prompt Builder → Step7 字段提取
→ Step8 Validation → Step9 Missing Detector → Step10 Insurance Mapper
→ Step11 输出标准JSON + 人工审核
```

### 12.1 分步开发与 JSON 契约

- 每一步做成**独立可测试的函数/服务**，输入输出严格用 **JSON 契约**约束。
- 便于分步开发、分步单测，禁止把多步逻辑揉成一个不可拆的黑盒。

### 12.2 AI 调用边界

- **AI 只做两件事**：文档分类（Step4）、字段提取（Step7）。
- 其余 9 步**禁止调用大模型**，全部用确定性本地代码实现，保证可复现、可测试、成本可控。

### 12.3 模型可替换

- 所有对大模型的调用必须封装在 `ai_service` 抽象接口内，例如：
  - `IDocumentClassifier`（Step4）
  - `IFieldExtractor`（Step7）
- **禁止**在业务代码里直接拼 Gemini / OpenAI API 请求，便于未来切换 GPT、Qwen 等。

### 12.4 字段状态三态（与第二节输出契约一致）

- 任何提取字段必须是 `extracted` / `missing` / `low_confidence` 之一。
- **禁止**用空字符串 `""` 代替「缺失」。

### 12.5 溯源保留

- OCR 阶段的 `bbox`、`page`、`confidence` 必须贯穿到最终输出，供人工审核在 PDF 上定位原文。

### 12.6 禁止模型猜测（Prompt 硬约束）

- 所有 Prompt 必须显式声明：**「没有依据的信息一律标记为 `missing`，不允许编造。」**

### 12.7 当前模型分工

| 步骤 | 模型/引擎 | 说明 |
|------|-----------|------|
| Step4 文档分类 | Gemini 2.5 Flash（轻量） | 仅判断文档属性，不提取字段 |
| Step7 字段提取 | Gemini 3.1 Pro Preview（精细） | 按标准字段输出结构化 JSON |

## 十三、代码模块约定

| 模块路径 | 职责 |
|----------|------|
| `apps/api/src/modules/pdf_extraction/steps/` | 各 Step 独立实现（`step1_upload.py` …） |
| `apps/api/src/modules/pdf_extraction/schemas.py` | 各 Step JSON 输入输出契约 |
| `apps/api/src/modules/pdf_extraction/ai_service/` | `IDocumentClassifier` / `IFieldExtractor` 及模型实现 |
| `apps/api/src/db/models/extraction.py` | `extraction_task` 等持久化模型 |

医生端测试入口：`/doctor/extraction-test`（左 PDF 预览 + 右标准字段可编辑，Step11 复用已有审核交互）。
