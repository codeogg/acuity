## 一、数据模型（PostgreSQL DDL，最终版）

> 相比早期草案，本版全部改为PostgreSQL语法：`BIGSERIAL`主键、`TIMESTAMPTZ`时间字段、`JSONB`替代JSON、`updated_at`统一用触发器自动更新。

### 1.1 通用触发器（自动更新 updated_at，所有表复用）

```sql
CREATE OR REPLACE FUNCTION trigger_set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### 模块 2：保单模板解析与标注（核心）

#### 状态机
```
PENDING → AUTO_PARSED → AI_ASSISTED → ANNOTATED → PUBLISHED
                                                  ↘ PARSE_FAILED(任意阶段解析异常可进入)
```

#### 2.1 模板上传
```
POST /api/admin/templates
Content-Type: multipart/form-data
Body: { company_id: number, template_name: string, file: File }
Response: { id, parse_status: "PENDING" }
```
**处理流程：**
1. 校验文件类型为PDF，大小 ≤ 20MB
2. 上传至对象存储，写入 `policy_template`（`template_code`若未提供则自动生成）
3. 通过 `arq` 入队异步任务 `parse_template_task(template_id)`
4. 立即返回，前端轮询或WebSocket获取解析进度

#### 2.2 自动解析任务（异步，arq worker执行）
**输入**：`template_id`
**逻辑**：
```
1. 加载PDF，判断是否含AcroForm
   - 有 → 逐个提取表单域(名称/类型/坐标) → recognize_source=AUTO_PDF, confidence=100
   - 无 → 逐页转图片 → 版面分析(文本块+空白区域启发式) → confidence按算法评分
2. 写入 template_field 表
3. 更新 policy_template.parse_status = AUTO_PARSED
4. 判断是否需要AI辅助：存在confidence<70的字段 或 为扫描件类型
   → 若需要，继续入队 ai_assist_task(template_id)
```
**失败处理**：PDF损坏/加密无法解析 → `parse_status = PARSE_FAILED`，写入失败原因到独立字段或日志，前端展示错误+提供重新上传入口

#### 2.3 AI辅助识别任务（异步）
**输入**：`template_id`
**逻辑**：
```
1. 获取低置信度字段所在页面 → 渲染为图片
2. 调用 Gemini(gemini-2.5-pro, Vertex AI) 视觉识别接口
   - 传入：页面图片 + 已有候选字段坐标(辅助定位) + 标准字段库列表(供AI给出映射建议)
   - 返回：字段标签、类型、坐标、AI建议的standard_field_code(仅作参考，不自动采信)
3. 新增/更新 template_field 记录，recognize_source=AI_VISION
4. 更新 policy_template.parse_status = AI_ASSISTED
```

#### 2.4 模板标注工具接口
```
GET    /api/admin/templates/{id}/fields                       获取所有字段(含映射状态)
POST   /api/admin/templates/{id}/fields                       人工新增字段(框选)
PUT    /api/admin/templates/{id}/fields/{fieldId}              编辑字段(坐标/类型)，需传row_version做乐观锁校验
DELETE /api/admin/templates/{id}/fields/{fieldId}
POST   /api/admin/templates/{id}/fields/{fieldId}/mapping      保存字段映射
  Body: { standard_field_id?, fixed_value?, checkbox_map_value?, transform_rule_id? }
POST   /api/admin/templates/{id}/preview-fill                  测试填充预览
  Body: { sample_values: {field_code: value, ...} }
  Response: { preview_pdf_url }
POST   /api/admin/templates/{id}/publish                       发布模板
```

**乐观锁并发控制（`PUT .../fields/{fieldId}`）：**
```python
# service层伪代码
async def update_field(field_id: int, data: FieldUpdateDto, expected_version: int):
    result = await db.execute(
        update(TemplateField)
        .where(TemplateField.id == field_id, TemplateField.row_version == expected_version)
        .values(**data.dict(), row_version=TemplateField.row_version + 1)
    )
    if result.rowcount == 0:
        raise ConflictException("该字段已被他人修改，请刷新后重试")
```

**发布校验逻辑：**
```python
async def publish_template(template_id: int):
```

### 模块 3：AI病历识别

#### 3.1 提取接口
```
POST /api/doctor/ai/extract
Body: { medical_record_text: string, template_id: number }
Response: {
  extracted_fields: {
    "patient_name_cn": {"value": "陈大文", "confidence": 0.92},
    "diagnosis": {"value": "上呼吸道感染", "confidence": 0.88},
    ...
  },
  process_time_ms: 1820,
  token_usage: 456
}
```

**处理逻辑：**
```python
async def extract(text: str, template_id: int, clinic_id: int) -> ExtractResponse:
    # 1. 限流检查(Redis) - 每诊所每分钟最多N次
    await check_rate_limit(clinic_id)

    # 2. 通过template_field_mapping反查该模板需要哪些standard_field
    required_fields = await get_required_fields_by_template(template_id)

    # 3. 动态构建Prompt + JSON Schema
    prompt, schema = build_extraction_prompt(text, required_fields)

    # 4. 调用Gemini(超时10s，失败重试2次，指数退避)
    try:
        result = await gemini_client.extract_structured(prompt, schema)
    except GeminiTimeoutError:
        raise AiServiceUnavailableError("AI识别暂时不可用，请手动填写")

    # 5. 校验返回值与validation_rule
    validated = validate_fields(result, required_fields)
    return validated
```

**关键规则：**
- Prompt中明确要求"找不到依据时value填null，禁止编造"
- `confidence < 0.6` 的字段前端需高亮标黄，提示医生重点核对
- AI服务不可用时，不阻塞流程，允许医生直接手动填写（`ai_raw_result`记为null）

#### 3.2 同患者多模板复用
```
POST /api/doctor/claims/{submissionId}/reuse-for-template
Body: { new_template_id: number }
Response: { submission_id: number, prefilled_fields: {...} }
```
**逻辑**：从原submission的`final_field_values`中，按新模板所需字段取交集预填，缺失字段标记需要重新提取

```
Response: { pdf_url: string, generated_at: string }
```

**填充引擎逻辑（核心）：**
```python
async def generate_filled_pdf(submission_id: int) -> str:
    submission = await get_submission(submission_id)
    mappings = await get_field_mappings(submission.template_id)  # join template_field
    pdf_doc = fitz.open(await download_original_pdf(submission.template_id))

    missing_required = []
    for m in mappings:
        value = resolve_field_value(m, submission.final_field_values)  # fixed_value或标准字段值
        if value is None and m.standard_field.is_required:
            missing_required.append(m.standard_field.field_code)
            continue
        if m.transform_rule:
            value = apply_transform(value, m.transform_rule)
        draw_field_on_pdf(pdf_doc, m.template_field, value, submission.doctor_id)

    if missing_required:
        raise ValidationException(f"以下必填字段缺失: {missing_required}")

    pdf_bytes = pdf_doc.tobytes()
    url = await upload_to_storage(pdf_bytes, f"generated/{submission.submission_no}.pdf")
    await update_submission(submission_id, generated_pdf_url=url)
    return url
```

**字段类型绘制规则：**
| field_type | 处理方式 |
|---|---|
| text | 按`font_size`绘制文本，超出`width`时自动缩小字号(最小6pt)，仍超出则截断+日志告警 |
| checkbox/radio | 值等于`checkbox_map_value`时绘制勾选标记 |
| signature | 从`doctor.signature_url`加载图片绘制 |
| image | 从字段值(图片URL)加载绘制 |
| date | 按`transform_rule`格式化后绘制文本 |

#### 状态机
```
DRAFT → AI_FILLED → CONFIRMED → PRINTED
  ↓         ↓            ↓
      CANCELLED(任意阶段可作废，需二次确认)
```

#### 接口清单
```
GET    /api/doctor/insurance-companies                          仅返回当前诊所已配置的保司

POST   /api/doctor/claims                                       创建草稿 { company_id, template_id }
PUT    /api/doctor/claims/{id}/medical-record                   提交病历文本，触发AI识别，status→AI_FILLED
PUT    /api/doctor/claims/{id}/fields                            医生确认/修改字段值
  Body: { final_field_values: {field_code: value} }
  副作用: diff对比ai_raw_result，写入claim_field_change_log
POST   /api/doctor/claims/{id}/confirm                            校验必填完整性，status→CONFIRMED
POST   /api/doctor/claims/{id}/generate-pdf                       见模块4

POST   /api/doctor/claims/{id}/cancel                              status→CANCELLED

GET    /api/doctor/claims/{id}
```

**数据隔离强制规则（所有 `/api/doctor/**` 接口通用）：**
```python
# FastAPI依赖注入，所有doctor路由必须使用此依赖
async def get_current_doctor(token: str = Depends(oauth2_scheme)) -> CurrentDoctor:
    payload = decode_jwt(token)
    if payload["role"] != "DOCTOR":
        raise HTTPException(403)
    return CurrentDoctor(id=payload["sub"], clinic_id=payload["clinic_id"])

# service层查询自动带clinic_id过滤，禁止跨诊所访问
async def get_claim(claim_id: int, clinic_id: int):
    claim = await db.get(ClaimSubmission, claim_id)
    if not claim or claim.clinic_id != clinic_id:
        raise HTTPException(404)  # 故意返回404而非403，不暴露资源存在性
    return claim
```

## 三、关键异常与边界处理清单

| 场景 | 处理策略 |
|---|---|
| PDF为扫描件且OCR/AI识别质量差 | 允许标注员完全手动框选补充，不强制依赖自动识别 |
| Gemini返回不合法JSON | SDK层用response_schema强约束，理论上不会出现；仍需try/except兜底+重试2次 |
| 模板使用中被要求修改 | 无历史记录直接编辑；有历史记录强制走新版本流程(见2.4发布逻辑) |
| 填充内容超出PDF字段空间 | 自动缩小字号(最小6pt)，仍超出则截断+记录告警供后续调整框体 |
| 标注并发冲突 | `template_field.row_version`乐观锁，冲突返回409+提示刷新 |
| AI服务超时/不可用 | 不阻塞医生流程，允许手动填写，前端明确提示 |
| 跨诊所越权访问 | 依赖注入层强制clinic_id过滤，返回404而非403 |
