# 信息提取验收标准（判定“通过”的条件）

本文件规定 CMS 病历数据到理赔表单的信息提取系统，在本测试集上判定为通过所需满足的标准。每份 PDF 的输出均对照 [`key.json`](ground_truth.json) 评分。

> **原则性目标（不可违背）** 见 [`EXTRACTION_PRINCIPLES.zh-CN.md`](EXTRACTION_PRINCIPLES.zh-CN.md)。

本测试集在语言、版式、扫描质量、数据完整度上刻意做了混合，只能应付理想情况的系统会明显暴露问题。合格系统应做到：能提取的照实提取；提取不到的如实标记为缺失，绝不臆造；忽略占多数的干扰信息；且绝不把数据绑定到错误的病人身上。

---

## 一、标准目标字段

每份导出文件都对照香港各保险公司理赔表单（住院 / 手术 / 预授权）所需字段的并集评分。系统必须对每个字段给出结果：要么是标准化后的取值，要么明确标记为 `missing`（并说明原因）。

| 字段 | 标准化格式 | 说明 |
|-------|-----------------|-------|
| `patient_name` | 按原文（中文和/或英文） | 中英文任一即可；两者都有则都保留 |
| `dob` | `YYYY-MM-DD` | |
| `gender` | `Male` / `Female` | 对 男/女 做映射 |
| `hkid` | `A123456(7)` | 保留打码形式（`A12XXX6(7)`），不得猜测被遮蔽的数字 |
| `patient_phone` | 香港 8 位号码 | |
| `insurer_name` | 自由文本 | |
| `policy_number` | 按原文 | 不得与会员号 / 凭证号 / 网络代码混淆 |
| `member_cert_no` | 按原文 | |
| `clinic_name` | 中文和/或英文 | 医院或诊所 |
| `doctor_name` | 中文和/或英文 | 若与注册编号连在一起，须拆分为独立字段 |
| `admission_date` | `YYYY-MM-DD` | |

## 二、功能要求

### （一）文件接入与版式
1. 同时支持带文本层的 PDF 和纯图像 PDF（无文本层），并能自动判别收到的是哪一种（不得默认存在文本层）。
2. 支持多页文档；单个字段或表格可能跨页（文档 11、12）。
3. 支持单栏、双栏、表格、发票、信函 / 正文段落、传真封面等多种版式。

### （二）OCR（针对扫描件——文档 03、04、07、08、11、12）
4. 对无文本层的纯图像页面进行 OCR。
5. 识别前先检测并校正页面方向：90°（文档 04）和 180°（文档 08）。
6. 能容忍轻微倾斜（±2°）、噪点、对焦不实、低对比度（文档 07），以及严重的 JPEG 压缩 / 低分辨率劣化（文档 12）——此类情况应相应下调置信度，而不是输出乱码。
7. 能读取盖章和手写批注，且不干扰相邻的印刷字段（文档 04）；手写内容不要求转写，但不得被误当作字段取值。

### （三）语言
8. 能从纯中文（05–08）、纯英文（01–04）、中英混合（09–12）文档中提取信息。
9. 能处理跨语言的“标签—取值”搭配：英文标签配中文取值，或反过来（文档 09）。
10. 将中英双语字段标签映射到同一标准字段（診斷 = Diagnosis、病房 = Ward、手術 = Procedure 等）。

### （四）字段映射与标准化
11. 将原文标签（中英文均可）映射到第一节的标准字段。
12. 将各种日期写法统一为 ISO 格式：`DD/MM/YYYY`、`DD-MM-YYYY`、`DD MMM YYYY`、`YYYY年M月D日`，以及存在歧义的 `MM/DD/YYYY`（文档 02）——须给出判定或标记歧义，不得默默按错误方式解析。
13. 将金额（`HK$`、`HKD`、千分位分隔符）统一为“数值 + 币种”。
14. 保留身份证号的原始格式及打码（文档 05）——打码的身份证号属于不完整取值，不得补全。
15. 医生姓名与注册编号连排时，须将两者拆开（文档 07、04）。

### （五）缺失识别（核心要求）
16. 文档中不存在的每个标准字段，一律返回 `missing`——不得编造、推断，或从别的文档搬取。
17. 反例文档（文档 12）仅含 `patient_name` 和 `diagnosis_text`；系统必须返回这两项，其余 17 项标记为缺失。臆造这 17 项中的任意一项，直接判不通过（见第四节）。
18. 区分“字段本就不存在”与“字段存在但 OCR 置信度低”。

### （六）干扰信息剔除
19. 下列噪声一律不得映射到任何理赔字段：排队号 / 取号、预约编号、下次预约日期、会员 / 积分及等级、营销同意标志、获知渠道、病历卡颜色代码、诊室 / 床位号、网络 / 路由代码、条形码、药房 SKU、打印人 / 打印时间、诊所营业时间、访客 WiFi、评价星级、天气，以及各类模板 / 隐私声明（PICS）文字。
20. 以下混淆尤须避免：积分 ≠ 会员号；网络代码 / 系统流水号 / 条形码 ≠ 保单号；预约编号 ≠ 收据号；下次预约日期 ≠ 就诊 / 入院日期；打印人 ≠ 医生 / 病人姓名。

### （七）消歧与冲突处理
21. 一份文档含多次就诊（文档 02）：须绑定到正确的那次（住院 / 手术那次），而不是用于充数的普通科 / 药房记录行。
22. 一份文档含多名病人（文档 08）：所有字段必须绑定到目标病人（文档中标注为“索償對象”），绝不能落到一同打印的干扰病人身上。绑错病人直接判不通过。
23. 取值冲突（文档 10：最终诊断 vs 初步诊断；文档 02、06：打印总额 ≠ 各明细之和）：须按规则取权威值（最终优先于初步；对总额与明细做核对或标记），并把冲突显性提示出来，而不是默默选一个了事。
24. 跨页汇总（文档 11）：总额需要把两页的明细项相加得出。

### （八）置信度、来源与输出
25. 为每个字段输出置信度；来自 OCR、打码、低对比度、存在冲突的字段，其置信度必须低于来自干净文本层的字段。
26. 为每个字段输出来源信息（页码，条件允许时给出边界框或原文标签），便于人工复核。
27. 当置信度低、检测到冲突，或理赔关键字段缺失时，将相应文档 / 字段标记为需人工复核。
28. 输出结构化结果（如 JSON），以标准字段为键，包含 `value`、`status`（`extracted` / `missing` / `low_confidence`）、`confidence`、`source`。

### （九）就诊选择与全量病历甄别（文档 13–16）
29. 一份文档含多次就诊时（选择器 15、16；文档 13、14 中的就诊历史），须列出全部就诊并支持选定其中一次；提取范围只限所选的那次就诊。
30. 对选定的就诊，只返回该次就诊的字段——输出中不得混入其他就诊的取值（不得跨就诊串用）。
31. 从全量病历导出（13、14）中，只提取本次理赔就诊的字段；不得把慢性病问题列表、既往诊断、长期用药、化验结果、疫苗接种、家族 / 社会史，或其他既往 / 无关就诊的内容并入本次理赔。
32. 区分本次理赔诊断（就诊相关）与问题列表上的活动性慢性病——例如阑尾炎 / 胆石症这次发作，与病人本身的糖尿病 / 高血压背景要分清。
33. 同一次就诊在多处出现时（如文档 13 的就诊表格行与就诊明细段），须核对并归并为同一套取值。

## 三、量化通过门槛

逐字段对照 `key.json` 评分，并汇总全部 16 份文档。

| 指标 | 定义 | 通过门槛 |
|--------|------------|----------------|
| 已有字段**准确率** | 正确取值 ÷ 输出取值 | ≥ 0.95 |
| 已有字段**召回率** | 正确取值 ÷ 标准答案中实际存在的取值 | ≥ 0.90（文本层）、≥ 0.80（扫描 / OCR） |
| **关键字段**召回率 | 针对 `patient_name`、日期基准、`diagnosis_text`、`procedure_text`、`amount_total`，以及存在时的保险公司 / 保单 | ≥ 0.95 |
| **缺失识别**召回率 | 正确标为缺失的字段 ÷ 实际缺失字段 | ≥ 0.95 |
| **干扰信息剔除** | 未被映射到任何字段的干扰项 ÷ 干扰项总数 | = 1.00（任何干扰项都不得填入字段） |
| **日期标准化**准确率 | 正确统一为 ISO 的日期 ÷ 出现的日期 | ≥ 0.95 |
| 置信度**校准** | 扫描 / 打码 / 冲突字段的分值低于干净字段 | 必须满足（单调） |

某取值经标准化后与标准答案一致，即计为**正确**（日期按 ISO；金额按数值；姓名按中英文任一匹配；编码须完全一致；诊断 / 手术等自由文本按标准化后的子串或语义匹配）。

**单份文档通过：** 关键字段全部正确、无干扰项被填入字段、未触发任何一票否决项（第四节），且缺失字段全部如实标记为缺失。

**整体通过（PASS）：** 以上七项指标全部达标，全测试集零一票否决，且至少 14 / 16 份文档单独通过。

## 四、一票否决项（无论得分高低，直接判不通过）

在理赔场景下，下列情况不可接受，一旦出现，本次运行直接判不通过：

- **绑错病人**——文档 08 中任何字段取自干扰病人。
- **臆造取值**——文档 12 中 17 个不存在的字段被给出取值；或任何字段填入了该文档中根本没有的数据。
- **干扰项被填入字段**——如把积分当会员号、把网络代码当保单号、把预约编号当收据号。
- **歧义日期被默默解析错**——文档 02 的 `MM/DD/YYYY` 被高置信度地统一成错误日期，且未标记歧义。
- **打码身份证号被补全**——文档 05 被遮蔽的数字被臆造出来。
- **文档 10 的两个冲突诊断未加区分**——即把初步诊断当作最终诊断高置信度输出。
- **跨就诊串用**——把未选定就诊的字段当成本次理赔的（文档 15、16），或在全量病历中拿非目标就诊的数据充当本次理赔（文档 13、14）。
- **既往 / 慢性数据并入本次理赔**——把背景慢性病（糖尿病 / 高血压）或既往无关就诊的诊断 / 手术当作本次理赔的（文档 13、14）。

## 五、评分流程

1. 用全部 16 份 PDF 跑一遍流程，逐份记录结构化输出。
2. 逐份文档，把输出字段与 `key.json` 中的 `present_fields`、`missing_required_fields` 对齐。
3. 统计：正确的已有字段、错误的已有字段（准确率失分）、漏提的已有字段（召回率失分）、正确标缺失、该标缺失却未标、干扰项被填入、一票否决项。
4. 计算第三节各项指标；核对第四节的一票否决项。
5. 按单份文档和整测试集两级给出通过 / 不通过结论，并逐一列出未达标字段。
