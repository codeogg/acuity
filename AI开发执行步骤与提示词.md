# 运营模拟（Impersonation）模块 —— AI Coding 执行步骤与提示词

配合《运营模拟功能详细设计文档.md》使用。建议每一步单独开一轮对话/任务给 AI coding 工具（如 Claude Code），不要一次性把整个模块丢给它，原因见"使用说明"。

---

## 使用说明（先读）

1. **先把设计文档喂给它，再分步开发**，不要指望 AI 一次性生成整个模块。模拟功能涉及数据库、鉴权、中间件、前端多端联动，一次性生成的代码大概率会在细节上（比如超时兜底、令牌一次性校验）偷工减料。
2. **每一步做完，先自己（或让 AI）跑测试再进入下一步**，尤其是权限拦截和审计日志这两步，是安全底线，必须验证到位再往下走。

3**每一步的提示词里都要求它"先说明实现方案，再写代码"**，避免它跳过设计直接写、埋下架构错误。
4. 提示词里出现的 `[你的技术栈]`、`[表名/路径]` 等占位符，使用前替换成你们项目实际情况（如 Java+MyBatis / Node+Prisma / 表名前缀等）。

## 步骤 5：权限控制注解与拦截中间件（安全核心，单独一步，务必仔细）

**提示词：**

```
基于设计文档第六章"权限控制"，实现以下内容：
1. 定义 @ImpersonationAccess 注解，可选值 READ_ONLY / MUTATING / FORBIDDEN，
   默认值为 MUTATING（即未标注时按 MUTATING 处理，这是安全底线，不能改）
2. 实现一个切面/中间件，逻辑如下：
   - 判断当前请求上下文里是否存在 impersonation_context（第4步生成的 token 解出）
   - 不存在：直接放行，不做任何处理
   - 存在：读取目标接口的 @ImpersonationAccess 值
     - FORBIDDEN → 直接拒绝，无论 mode 是什么
     - MUTATING → mode=view 时拒绝，mode=proxy 时放行
     - READ_ONLY → 放行
3. 请特别说明：如果某个接口完全没有加这个注解，你的代码会怎么处理
   （期望答案：按 MUTATING 处理，即检视模式下会被拒绝）
4. 写 3 个测试用例：
   - 一个标了 FORBIDDEN 的接口，检视和代理模式下都应该被拒绝
   - 一个标了 MUTATING 的接口，检视模式拒绝、代理模式放行
   - 一个完全没标注的接口，检视模式下应该被拒绝（验证默认拒绝生效）

这一步是安全边界，请先写出切面的完整判断逻辑伪代码，我确认无误后再落地成正式代码，
并且必须带上面这 3 个测试用例一起提交。
```
